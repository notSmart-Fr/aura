## Background Worker Caching and Volatile Context Hydration

To ensure self-healing fault tolerance and guarantee that customers receive the most up-to-date pricing and inventory availability, the storefront queue worker (`scripts/worker.ts`) transitions away from static, monolithic response caching to a **Volatile Layered Context Hydration Pattern**.

### Architectural Design

Instead of caching the final text generated by the LLM agent (which risks locking in transient failures and returning stale data), the worker executes a fresh execution run for every inbound event. The pipeline utilizes a shared database connection pool to balance performance and avoid socket exhaustion.

```text
[ BullMQ Ingest ] ──► [ TypeORM Bootstrapped Pool ] 
                             │
                             ├──► (Passes master pool socket) ──► [ Kysely Engine ]
                             │                                           │
                             │                                    (Vector Match)
                             │                                           ▼
                             └─► [ Live Hydration ] ◄─────── [ Product IDs / Names ]
                                  (Price/SKU/Enabled)
                                         │
                                         ▼
                               [ Gemini / DeepSeek ] ──► [ WhatsApp API HTTP Post ]
```

### 1. Zero-Overhead Connection Sharing

During initialization, the worker bootstraps a TypeORM `DataSource` using database environment variables. To prevent socket duplication and thread exhaustion, the underlying Postgres client pool (`typeormDataSource.driver.master`) is wrapped directly into a `Kysely` instance.

### 2. Layered Resolution Flow

1. **Stable Semantic Lookup**: The worker computes the vector coordinates of the inbound query and delegates similarity matching to `Kysely`. Kysely performs a type-safe `pgvector` distance lookup (`<=>`) against the `product` and `product_translation` tables to extract stable, slow-changing database entity identifiers (Product IDs, Names, Slugs).
2. **Volatile Context Hydration**: Immediately after Kysely resolves, the worker queries the core `product_variant` table directly using the shared pool to retrieve volatile, fast-changing transactional columns (prices, SKUs, and availability/enabled flags).
3. **AI Grounding**: The volatile variant metrics are constructed into a structural grounding context block and appended to the agent's prompt to ensure all generated replies reflect real-time storefront state.
4. **Fault Propagation**: No part of the final generated string or transient errors is saved to a cache database. Pipeline failures (Gemini rate limits, timeouts) propagate naturally, triggering standard BullMQ job retries.

Database and integration failures at cache boundaries are wrapped in typed domain errors (`DatabaseDomainError`, `IntegrationError`) — see [Boundary Exception Architecture](./boundary-exceptions.md).
